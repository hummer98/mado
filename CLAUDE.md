# mado — AI Development Policy

CLI-first Markdown viewer built with Electrobun (Bun + WKWebView).

## プロジェクト概要

```
mado README.md          # ネイティブウィンドウで開く（Hot Reload 付き）
mado https://...        # URL を開く
mado render file.md     # PNG を出力（AI 向け画像変換）
```

詳細コンセプト: `docs/seed.md` を参照。

## コーディング規約

### 言語方針

- **ドキュメント・コメント**: 日本語
- **コード（変数名・関数名・型名・定数）**: 英語
- **コミットメッセージ**: 英語（imperative mood）

### TypeScript

- strict モードを常に有効にする（`"strict": true`）
- `any` は原則禁止。型が不明な場合は `unknown` を使い、適切に絞り込む
- スキーマ検証には **Zod** を使用する。外部入力（CLI 引数・ファイル・URL レスポンス）は必ず Zod でバリデーションする
- `as` によるキャストは最終手段。型ガードで代替できる場合はそちらを優先する
- `import type` を型インポートに使用する
- 関数の返り値型は明示する（推論に任せない）

### ファイル構成

- 1ファイル1責務。副作用を持つコードはエントリポイントに集める
- 共通ユーティリティは `src/lib/` に置く
- 型定義は `src/types/` に置く（ただし小さな型は使う場所に置いてよい）

### エラーハンドリング

- エラーは `catch` で握りつぶさない。最低限ログを残す
- ユーザー向けエラーメッセージは日本語で分かりやすく
- 内部エラーは英語で詳細に（ログ・スタックトレース）

## ロギングポリシー

（cmux-team から移植）

### ログインターフェース

イベント名でレベルを区別する。

| イベント名パターン | 用途 | 例 |
|---|---|---|
| `error` | 操作失敗・例外 | `log("error", "file read failed: ...")` |
| `*_failed` | 特定操作の失敗 | `log("watch_failed", ...)` |
| `*_started`, `*_completed` | ライフサイクルイベント | `log("renderer_started", ...)` |
| その他 | 状態変化・判断記録 | `log("hot_reload_triggered", ...)` |

### 必ずログすべきイベント

1. **例外捕捉時**: `catch` で処理する場合、最低限 `log("error", ...)` で記録する
2. **外部リソースアクセス失敗**: ファイル読み込み・URL fetch の失敗
3. **Hot Reload の発火**: ファイル変更検知 → WebView 更新のサイクル
4. **状態遷移**: ウィンドウのオープン・クローズ

### 禁止事項

- **空の `catch {}`**: 必ずログを残す。ただし冪等な後処理（タブ閉じる等）は例外
- **高頻度ループ内の過剰ログ**: 変化があった場合のみ記録する
- **機密情報のログ**: 認証トークン等を含めない

### フォーマット

```
[2026-04-12T10:30:00+09:00] event_name key1=value1 key2=value2
```

- タイムスタンプはローカル TZ 付き ISO 8601
- 1 行 1 イベント

## テスト方針

### レンダリング層（marked + mermaid）

- Playwright + 通常ブラウザ（Chromium）でテストする
- WebView に依存しない純粋な JS ロジックのテスト

```bash
bun test:rendering   # Playwright でレンダリング層のみテスト
```

### アプリ統合テスト（Hot Reload 等）

Electrobun の `evaluateJavascriptWithResponse()` を使った自前 E2E 基盤を使用する。
詳細は `docs/seed.md` §E2E テスト戦略を参照。

```bash
bun test:e2e   # Electrobun アプリの統合テスト
```

### ユニットテスト

```bash
bun test       # bun:test ランナー
```

## GitHub Issue 運用

### Issue を作成すべき場面

- 実験中に発見した具体的な失敗パターン（再現手順付き）
- 設計判断が必要で複数の選択肢がある場合
- ドキュメントと実装の乖離

### Issue に含める情報

- **問題**: 何が起きたか
- **原因**: なぜ起きたか（分かる場合）
- **修正内容**: 具体的な変更案
- **対象ファイル**: 修正が必要なファイル一覧

### Issue を作成しない場面

- typo やフォーマットの軽微な修正 → 直接コミット
- 明らかなバグ修正 → 直接コミット

## 判断基準

- **「動くか？」が最優先** — 理論的な美しさより実際に動作すること
- **実験で検証してから本実装** — プロトタイプで試してから本コードに反映
- **既存の動作を壊さない** — CLI インターフェースを安定させる
- **設計判断で迷ったら Issue を作成** してユーザーの判断を仰ぐ
